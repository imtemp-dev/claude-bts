# 경쟁자들은 Painkiller인가?

> 우리가 분석한 3개 프로젝트를 동일한 기준으로 평가한다.
> 핵심 질문: **사용자가 매일 쓰는가? 없으면 불편한가? 돈을 내는가?**

---

## 트래킹 지표 (2026-03 기준)

| 프로젝트 | GitHub Stars | Releases | 가격 | 유료 사용자 추정 |
|----------|-------------|----------|------|----------------|
| ContextStream | 30 | 103 | $0-40/seat/mo | 불명 (낮을 것) |
| memctl | 11 | 초기 | $0-149/mo | 불명 (매우 낮을 것) |
| claude-mem | AGPL, 비공개 | v10.5.6 | 무료 | 불명 |

**30 stars, 11 stars.** 이 숫자가 말해주는 것: **아직 아무도 painkiller를 찾지 못했다.**

103 releases를 했는데 30 stars라는 것은, 기능을 많이 만들었지만 사용자가 "이거다!"라고 느끼지 못했다는 의미다.

---

## 경쟁자별 평가

### claude-mem: "모든 것을 기억하는 AI"

**매일 푸는 문제**: 세션 시작 시 이전 맥락 자동 주입 (SessionStart hook)
**빈도**: 매일 (세션 시작마다)

**Painkiller 요소**:
- SessionStart에서 이전 observations + summaries 주입 → "어제 뭐 했더라?" 해결
- PostToolUse 120초 타임아웃으로 매 도구마다 캡처 → 빠짐없이 기록
- Claim-confirm 큐로 크래시 안전 → 데이터 유실 방지

**Vitamin 요소**:
- 35,650줄의 복잡성 → 설치/유지 부담
- 모든 도구 실행을 캡처 → 88% 노이즈 → 주입되는 컨텍스트 품질 불균일
- Worker 프로세스 + Agent 서브프로세스 + ProcessRegistry → 좀비/메모리 관리 필요
- Chroma 벡터 DB → 추가 인프라
- 3개 AI 프로바이더(Claude/Gemini/OpenRouter) → 설정 복잡성

**솔직한 평가**: SessionStart 주입이라는 **painkiller 기능이 있지만, 35K줄의 vitamin 인프라에 묻혀 있다.** "기억해 준다"는 가치보다 "설정이 복잡하다"는 비용이 더 크게 느껴질 수 있다.

**증거**: v10.5.6까지 갔는데도 Claude Code 생태계에서 주류가 되지 못함. Bun 의존성, AGPL 라이선스가 장벽.

### ContextStream: "AI 컨텍스트 + 지식 그래프"

**매일 푸는 문제**: 매 메시지마다 관련 컨텍스트 주입 (context tool)
**빈도**: 매 메시지 (가장 높은 빈도)

**Painkiller 요소**:
- per-message context injection → AI 응답 품질 향상
- Lesson 시스템 → 과거 실수 반복 방지 (RISKY_ACTION_KEYWORDS)
- Pre/post compaction → 컴팩션 후 맥락 복구
- 27개 훅 → 가장 포괄적인 lifecycle 커버리지

**Vitamin 요소**:
- **클라우드 의존** → 오프라인 불가, 매 요청마다 API 호출
- 42,486줄 → 7개 토큰 전략, 8개 에디터 지원 등 기능 과잉
- 176개 API 메서드 → AI가 "어떤 도구를 써야 하지?" 혼란 (75% 토큰 절감이 필요한 이유)
- 가격: $20-40/mo → "이 가치가 $40인가?" 질문에 답해야 함
- Free tier 5,000 operations (평생) → 빠르게 소진 후 유료 전환 압력

**솔직한 평가**: per-message injection은 **vitamin**이다. "응답이 약간 더 나아진다"는 가치로는 $20/mo를 정당화하기 어렵다. Lesson 시스템은 **painkiller에 가깝지만**, 수동으로 `capture_lesson`을 호출해야 하고 proactive injection은 하드코딩된 20개 키워드에 의존한다.

**증거**: 103 releases, 30 stars. 엄청나게 많은 기능을 만들었지만 사용자가 따라오지 않음. "more features = more value"가 아니라는 증거.

### memctl: "팀 공유 메모리"

**매일 푸는 문제**: 팀 수준 지식 저장/검색
**빈도**: 저장은 세션마다, 검색은 필요 시

**Painkiller 요소**:
- Org/Project/Role RBAC → 팀 접근 제어
- 하이브리드 검색 (FTS5 + 384-dim vector, RRF) → 검색 품질
- Intent 분류 (5 타입) → 쿼리에 맞는 검색 전략
- Health-based eviction → 자동 정리
- Session claims → 동시 편집 충돌 감지

**Vitamin 요소**:
- **캡처 품질이 낮음** → 키워드 기반 `extractHookCandidates()`. LLM 추출 없음
- 팀 전체 도입 필요 → 한 명이 쓰면 가치 없음
- Turso 클라우드 의존
- 28개 테이블 중 핵심은 ~8개. 나머지는 blog_posts, promo_codes 등
- Free에서 모든 기능 → 업그레이드 동기 약함 (용량 제한만)

**솔직한 평가**: **팀 도구인데 팀 도입 장벽이 높다.** 그리고 저장되는 내용의 품질(키워드 추출)이 낮아서, 나중에 검색해도 "이게 뭐지?"가 될 수 있다. 좋은 인프라(검색, RBAC) 위에 낮은 품질의 데이터가 쌓이는 구조.

**증거**: 11 stars, 2주 전 론칭. 너무 이르지만, 팀 도구는 cold start 문제가 심함.

## 공통 패턴: 왜 아무도 painkiller가 아닌가

### 1. "AI 메모리"라는 프레이밍이 모호하다

세 프로젝트 모두 "AI 메모리"를 표방한다:
- claude-mem: "memory compression system"
- ContextStream: "Smart Context & Memory"
- memctl: "Team shared memory"

**"메모리"는 목적이 아니라 수단이다.** 사용자는 "메모리"를 원하는 게 아니라, 구체적인 문제 해결을 원한다:
- "어제 하던 거 이어서 해" (세션 연속성)
- "이 코드 왜 이래?" (의사결정 추적)
- "이거 해봤어?" (탐색 공유)

"AI 메모리 도구"라고 하면 사용자는 "뭘 기억하는데? 왜 필요한데?"를 물어야 한다.

### 2. 모든 것을 기억하려 하면 아무것도 유용하지 않다

| 프로젝트 | 캡처 범위 | 결과 |
|----------|----------|------|
| claude-mem | 모든 도구 실행 | 88% 노이즈 → 검색/벡터 필요 |
| ContextStream | 매 메시지 | per-message API 호출 + 클라우드 비용 |
| memctl | 모든 hook 이벤트 | 키워드로 분류되는 저품질 데이터 |

**공통 문제**: 기억하는 양은 많은데, 기억하는 것의 **가치**가 불균일하다.

파일 읽기를 100번 기록해도, "왜 OAuth2를 선택했는가"라는 하나의 결정보다 가치가 낮다.

### 3. 설정 복잡성이 가치를 상쇄한다

| 프로젝트 | 설치 과정 |
|----------|----------|
| claude-mem | Bun 설치 → 플러그인 설치 → Worker 시작 → AI 프로바이더 설정 → Chroma 설정 (선택) |
| ContextStream | 가입 → API 키 → MCP 설정 → 에디터별 훅 설정 (2,169줄의 hooks-config) |
| memctl | 가입 → API 키 → MCP 설정 → 팀 초대 → 프로젝트 설정 |

**AI 코딩 도구를 쓰는 개발자가 원하는 것**: "설치하면 끝. 알아서 해줘."
**현실**: 설정 20분 + 동작 원리 이해 10분 + 문제 해결 30분.

## 그렇다면 시장에 painkiller 기회가 있는가?

### 있다. 하지만 다른 곳에 있다.

경쟁자들이 놓치고 있는 것:

**Pain #1: 세션이 끝나면 전부 잊는다 (매일)**
- claude-mem이 가장 가까이 갔지만 35K줄의 복잡성으로 덮어버림
- 가장 단순한 해결: 세션 끝에 요약 저장, 다음 세션에 주입. **5개 파일로 가능.**

**Pain #2: 컴팩션 후 맥락 날아간다 (주 2-3회)**
- ContextStream이 719줄(pre/post compact)을 투자해 해결
- 하지만 클라우드 의존. 로컬에서 30줄이면 해결 가능.

**Pain #3: PR description이 빈약하다 (주 2-5회)**
- **아무도 건드리지 않았다.** 3개 프로젝트 모두 PR 관련 기능이 0.
- 세션 중 축적된 decisions/constraints에서 PR description 자동 생성
- **가시적 결과물** → 바이럴 효과

## 결론

### 경쟁자들의 상태

```
claude-mem:     Painkiller 기능을 가진 Vitamin 제품 (복잡성이 가치를 상쇄)
ContextStream:  Vitamin 제품에 Painkiller 기능을 추가 중 (클라우드 의존이 장벽)
memctl:         Vitamin 인프라 위에 낮은 품질 데이터 (팀 도입 장벽)
```

### 시장 기회

```
30 stars + 11 stars + 비공개 = 아직 아무도 이기지 못한 시장.
이유: 모든 것을 기억하려다 아무것도 유용하지 않음.

기회: "적게 기억하되, 매일 쓸 수 있게" 하면
      3개 프로젝트 모두가 놓친 포인트를 잡을 수 있다.
```

### MVP에 대한 시사점

| 접근 | 빈도 | 파괴력 | 경쟁자 대비 |
|------|------|--------|------------|
| "모든 것을 기억" (경쟁자 방식) | 높음 | 낮음 | 30 stars 수준 |
| "why not만 기억" (우리 이전 방향) | 월 1-2회 | 중 | 차별화되지만 빈도 부족 |
| **"세션을 이어준다"** (수정 방향) | **매일** | **중-상** | **경쟁자보다 10배 단순하게** |
| **+ PR description 자동 생성** | **주 2-5회** | **상** | **아무도 안 함, 가시적 결과물** |

**가장 파괴적인 조합**:
1. **Day 1 가치**: 세션 연속성 (매일, 자동, 설치만 하면 끝)
2. **Week 1 가치**: PR description 자동 생성 (가시적 결과물, 바이럴)
3. **Month 3 가치**: 팀 decisions/constraints 축적 (lock-in)
